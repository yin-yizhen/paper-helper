# 浏览器就绪门

在任何 CNKI 检索前，按以下顺序自行取得可见页面，不要一上来要求用户手动开浏览器：

1. 读取应用内浏览器当前选中标签和标签列表，复用已打开的 CNKI 标签。
2. 再尝试接管用户浏览器中已连接的 CNKI 标签。
3. 仍没有时，在 Codex 应用内浏览器新建 CNKI 检索标签。
4. 只有上述三步都失败，才说明连接阻塞并让用户手动建立可接管会话。不要用 `Start-Process`、`cmd start`、后台 Playwright、裸 HTTP 或证书忽略参数跨过门槛。

| 状态 | 可做的事 | 下一步 |
| --- | --- | --- |
| 已连接且页面正常 | 检索、提取题录、下载 | 继续 CNKI 工作流 |
| 标签存在但工具未接管 | 再按上述顺序取得标签 | 三条自动路径均失败后才硬停止 |
| 418 / 证书错误 / 真正可见的验证码 / 登录页 | 仅保留已有台账 | 让用户在当前可见标签处理；通过后重新检查 |
| 无可见浏览器能力 | 仅报告阻塞条件 | 结束本次任务 |

页面正常的最低证据是能从可访问性快照或 DOM 中定位检索输入框/结果列表，并能读取当前 URL。不要把单一 CSS 选择器不存在误判成页面不可用。

验证码必须按“真正可见”判断：元素存在还不够；须同时满足 `display != none`、`visibility != hidden`、数值 `opacity > 0`、边界框宽高大于 0，且至少部分位于视口内。透明的 `#tcaptcha_transform_dy` 不得触发暂停。

**硬停止规则：** 自动取得页面失败，或当前确为 418、证书错误、登录/权限页、可见验证码时，不得生成题目、提纲、论文草稿、参考文献或 DOCX。恢复后从本门重新检查；用户说“继续”只是触发复检，不是通过证据。
