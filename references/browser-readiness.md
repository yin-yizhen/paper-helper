# 浏览器就绪门

在任何 CNKI 检索前，优先取得可见、可控的 **Chrome** 标签；不要一上来要求用户手动找浏览器：

1. 检查 Chrome 扩展会话，优先复用已打开的 Chrome CNKI 标签。
2. 若 Chrome 已连接但没有 CNKI 标签，在同一 Chrome profile 内新建标签并用 `tab.goto("https://kns.cnki.net/kns8s/search")` 导航。
3. 若 Chrome 已安装、扩展与 native host 可用但 Chrome 未运行，询问用户是否允许启动 Chrome；获同意后仅使用当前插件**明确提供**的 `open-chrome-window.js` 或等价启动器，再从第 1 步重试。
4. 只有 Chrome 不可用时，才把应用内浏览器作为题录检索的最后备选，并明确提示它不继承 Chrome 登录态和下载权限。

不得用 `Start-Process <URL>`、`cmd start`、系统默认浏览器、外部链接点击、后台 Playwright、裸 HTTP 或证书忽略参数跨过此门槛；它们可能把 CNKI 打开到 Edge 或不可控会话。

| 状态 | 可做的事 | 下一步 |
| --- | --- | --- |
| 已连接且页面正常 | 检索、提取题录、下载 | 继续 CNKI 工作流 |
| Chrome 标签存在但工具未接管 | 先复用/新建可控 Chrome 标签 | Chrome 路径失败后才考虑应用内浏览器兜底 |
| 418 / 证书错误 / 验证/登录页 | 仅保留已有台账与 Chrome handoff 标签 | 让用户在 Chrome 里的知网页签处理；通过后重新检查 |
| 无可见浏览器能力 | 仅报告阻塞条件 | 结束本次任务 |

页面正常的最低证据是能从可访问性快照或 DOM 中定位检索输入框/结果列表，并能读取当前 URL。不要把单一 CSS 选择器不存在误判成页面不可用。

验证码必须按双重规则判断：元素存在时，须同时满足 `display != none`、`visibility != hidden`、数值 `opacity > 0`、边界框宽高大于 0，且至少部分位于视口内；透明的 `#tcaptcha_transform_dy` 不得触发暂停。无论元素是否隐藏，只要**可见页面正文**出现“请完成安全验证”“向右滑动完成验证”或“拖动下方拼图完成验证”，也视为阻塞。

验证、登录、权限或证书阻塞时，必须将该 Chrome 标签前台可见并作为 `handoff` 保留。提示固定使用“Chrome 里的知网页签”；若当前前台是 `chrome://extensions` 等不可脚本页面，要求用户手动切换到标题为“安全验证”或 CNKI 的标签，不要继续自动操作。

**硬停止规则：** 自动取得页面失败，或当前确为 418、证书错误、登录/权限页、上述任一验证码证据时，不得生成题目、提纲、论文草稿、参考文献或 DOCX，也不询问论文参数。恢复后从本门重新检查；用户说“继续”只是触发复检，不是通过证据。
